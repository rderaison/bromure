# Bromure Remote — iOS / iPadOS Fat Client

Port of bromure-ac's **fat-client role** (REMOTE_FAT_CLIENT_PLAN.md) to iOS and
iPadOS: mirror a remote bromure-ac 1:1 — workspaces, terminals, coding-task and
automation boards, file manager — from an iPhone or iPad. Client only.

## Scope

**In:** connect flow (server directory, TOFU, password enrollment, P2P peers),
live mirror (0.75 s `/state` poll → the same mirror stores as macOS), terminals
(SwiftTerm), coding tasks + automations boards, remote file manager with Files
app integration, VM lifecycle / fusion / decision prompts.

**Out (by requirement):** running VMs on-device (so no browser pane — it *is* a
local VM on macOS), the system-wide network tunnel / SOCKS / utun ("VPN" —
later), being reachable from outside (client role only).

## Architecture

The macOS fat client's seams port unchanged; only the two process-shaped edges
are replaced:

| macOS | iOS |
|---|---|
| `SSHTunnel.dial` → `/usr/bin/ssh` subprocess over a socketpair | `SSHDialer` (FatClientSSHDial.swift): one **in-process NIOSSH connection per host**, one exec child channel per dial, bridged to a socketpair → same fd contract |
| `ssh-keygen` / `ssh-keyscan` / in-memory `ssh-agent` subprocesses | CryptoKit ed25519 identity in the same keychain item (`FatClientKeyStore`); NIOSSH handshake-capture host-key scan; pure-Swift known_hosts pin store |
| libghostty surface + `__attach-window` child process pump | SwiftTerm `TerminalView` + in-process pump speaking the same framed PTY protocol over `ControlClient.openStream` |
| `RemoteHostWindow` (NSWindow chrome) | SwiftUI navigation (iPhone: tab bar; iPad: split view) hosting the *same* content views (kanbans, dashboards, transcripts, sidebar data) |
| `NSAlert` decision prompts | `RemoteHostController.decisionPrompts` published state → SwiftUI alerts (same `/prompts/{id}/answer` wire) |

Everything else — `ControlClient` HTTP-over-fd framing, `RemoteHostController`
poll/reconcile, the five mirror stores (`SessionListModel`, `TabsModel`,
`GridLayoutStore`, `ScheduledAutomationStore`, `CodingTaskStore`), `Profile`,
the P2P broker/STUN/TURN stack, task/automation verbs — compiles on iOS as-is.

The iOS app is an Xcode project (`ios/BromureRemote.xcodeproj`, generated by
`scripts/gen-ios-project.py`) that compiles a **file subset** of
`Sources/AgentCoding` directly plus iOS-only sources in `ios/BromureRemote/` —
no SPM module split, no access-level churn. `ACAppDelegate` (the macOS app
delegate the task/automation *engines* reference) is a compile-only stub on
iOS; the engines run on the host, never on the client.

### File layout

- `Sources/AgentCoding/FatClientTypes.swift` — shared host/probe/keystore types (split from FatClientRemote.swift)
- `Sources/AgentCoding/FatClientSSHDial.swift` — in-process NIOSSH dialer (compiled+typechecked on macOS too)
- `Sources/AgentCoding/ControlClient.swift` — ControlClient + PTY frame codec (split from CLICommands.swift)
- `Sources/AgentCoding/SessionModels.swift` — TabsModel/SessionListModel + docker structs (split from the window files)
- `ios/BromureRemote/` — app entry, navigation, RemoteTransport (mobile), attach pump, SwiftTerm wrapper, connect/mirror/files screens
- `ios/BromureRemote.xcodeproj` — generated; regenerate via `python3 scripts/gen-ios-project.py`

### Files app integration

`UIFileSharingEnabled` + `LSSupportsOpeningDocumentsInPlace`: downloads land in
`Documents/<host>/<workspace>/…` and appear in Files under "Bromure Remote".
Uploads via `.fileImporter` (Files picker) → the same chunked
`POST /vms/{id}/file` op bridge; downloads offer Save-to-Files / ShareLink.

### Build

```bash
python3 scripts/gen-ios-sources.py     # refresh the shared symlink farm
python3 scripts/gen-ios-project.py     # regenerate the .xcodeproj
xcodebuild -project ios/BromureRemote/BromureRemote.xcodeproj -scheme BromureRemote \
  -destination 'generic/platform=iOS Simulator' CODE_SIGNING_ALLOWED=NO build
```

One-time: `xcodebuild -downloadComponent MetalToolchain` (SwiftTerm's shaders) and
an installed iOS simulator runtime. Device builds need a development team set in
Signing & Capabilities.

## Status (implemented + verified)

**Builds and runs.** macOS `bromure-ac` is unchanged (guards, not rewrites) — app
and test targets build; targeted tests pass (NativeTerminal/FatForward/GridLayout,
14 tests). The iOS package cross-compiles for `arm64-apple-ios-simulator`, the
generated Xcode project builds a real `BromureRemote.app`, and it launches on the
iOS 26.5 simulator (hosts list → connect → mirror).

Done:
- Cross-platform split of the shared subset (~55 AgentCoding files) behind
  `#if os(macOS)`; new shared files: FatClientTypes, SessionModels, ControlClient
  (+ PTYFrame codec), SharedStatusViews, FatForward, ConsentCredentialID,
  PlatformShims.
- In-process NIOSSH dialer (`FatClientSSHDial.swift`) + iOS `RemoteTransport`
  (`RemoteTransportMobile.swift`) — same static API as macOS, CryptoKit identity,
  NIOSSH host-key scan, pure-Swift known_hosts pin, classified probe.
- SwiftTerm terminal + in-process framed-PTY pump (`AttachSession.swift`) with
  reconnect/backoff; consent routed to iOS alerts via the controller's
  `decisionPrompts` (same `/prompts/{id}/answer` wire).
- App shell: RootView (adaptive stack/split), ConnectScreen (over the shared
  `RemoteConnectModel`), HostMirrorScreen (reused `RemoteHostController` +
  Coding/Automation boards), WorkspaceScreen (terminals / VM dashboard / Docker /
  Files). Files app integration via Info.plist (`UIFileSharingEnabled`,
  `LSSupportsOpeningDocumentsInPlace`) + `FileBrowserModel.lastDownloaded`.
- `scripts/gen-ios-sources.py` (symlink farm) + `scripts/gen-ios-project.py`
  (hand-emitted pbxproj, no xcodegen needed).

Deferred (matches scope): the system-wide network tunnel / SOCKS / utun (the
"VPN" — Phase 4 networking), the local-Chromium browser pane (it is a VM), and
the Trace Inspector UI (fetch methods are guarded mac-only for now).

Follow-ups worth doing next: a QuickLook/ShareLink presenter bound to
`FileBrowserModel.lastDownloaded`; a `.fileImporter` upload path into the same
`/vms/{id}/file` op bridge; an app icon + asset catalog; a compact iOS profile
settings screen over the `[String:Any]` profile-doc round-trip.

## Signing (required for the keychain)

The device identity + SSH client key live in the keychain, and iOS refuses every
`SecItemAdd` without a keychain entitlement (`errSecMissingEntitlement`, -34018)
— an unsigned build can't connect. The app therefore signs with a team:

- `App/BromureRemote.entitlements` declares `keychain-access-groups =
  $(AppIdentifierPrefix)io.bromure.remote`.
- The project uses automatic signing (`CODE_SIGN_STYLE = Automatic`,
  `DEVELOPMENT_TEAM = W3RD8G85BC`). Build with `-allowProvisioningUpdates`; for
  the simulator this resolves to "Sign to Run Locally" with the team-prefixed
  entitlement, which the simulator honors (verified: `SecItemAdd` → 0).
- Ad-hoc / unsigned simulator builds do NOT work for the keychain (both the
  data-protection and legacy keychains return -34018), so `CODE_SIGNING_ALLOWED=NO`
  is only viable for a pure compile-check, never for running a real connect.

Set `DEVELOPMENT_TEAM` (in `scripts/gen-ios-project.py`) to your own team for a
device build; the URL scheme + entitlement are otherwise portable.
